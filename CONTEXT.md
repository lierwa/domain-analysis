# domain-analysis（Social Intelligence MVP）

本仓库实现低成本社媒监听与洞察：用 **Topic** 组织监控项目，用 **Query** 配置关键词与数据源，采集公开内容后经清洗与分析输出看板与报告。本文档只收「产品与领域」用语，不收实现细节。

## Language

**Topic**:

一个监控/分析项目容器（对应 PRD 中的「项目 / Topic」），包含语言、市场等元数据；其下挂多条 **Query**。

_Avoid_: 用「项目」单独指代时不加语境（代码里 `topic` 与口语「项目」应对齐为 Topic）。

**Query**:

隶属于某一 Topic 的关键词组配置（包含词、排除词、平台、单次上限等），用于驱动采集。

_Avoid_: 与搜索引擎「查询字符串」混淆；在 UI 里若写「关键词组」须在 glossary 中视为 Query 的展示名。

**Crawl task（采集任务）**:

针对某一 **Query** 在某一数据源上的一次执行实例；有生命周期状态，用于异步跟踪采集进度与结果统计。

**Raw content（原始内容）**:

一次采集写入库中的一条公开帖子（或等价 UGC）记录，**尚未**经过清洗管道或 AI 结构化；用户应能按 Topic / Query 浏览以校验关键词与源站质量。在 **Topic** 级列表中，若多 **Query** 命中同一条站外帖子，**默认合并为一行**并展示命中 Query；用户可打开「按 Query 展开重复行」查看与调参。

_Avoid_: 把「任务」与「帖子」混称——前者是 **Crawl task**，后者是 **Raw content**。

**External ID（站外唯一标识）**:

平台为单条内容分配的稳定 ID（例如 Reddit 帖子的 `t3_…` / `name`，X 推文的数字 `id`）。用于判断「是不是同一条站外帖子」，从而做去重或 Topic 视图下的合并行；若某源未提供，则只能退化为较弱信号（如规范化 URL），合并策略需在实现里单独说明。

_Avoid_: 对用户界面文案直接使用字段名 `external_id`；可展示为「帖子 ID」或仅在调试视图出现。

## Relationships

- 一个 **Topic** 包含多条 **Query**。
- 一条 **Query** 可产生多个 **Crawl task**（不同平台、多次手动/周期运行）。
- 一个 **Crawl task** 成功后可产生多条 **Raw content**（条数受 limit 与源站结果约束）。

## Example dialogue

> **Dev:**「采集任务显示 success 之后，用户第一步应该看到什么？」  
> **Domain expert:**「先看到 **Raw content** 列表，确认抓到的帖子和关键词是否匹配；确认后再谈清洗和 AI 标签。」

## Flagged ambiguities

- （暂无）后续若「Topic」与阶段 2 的 Brand/Competitor 并存，再拆开术语。
- **合并行 vs 存储**：列表上的「合并一行」是 **Topic 视图** 的呈现；持久层仍可一条 **Raw content** 一行（按 `query_id` 等区分）。若实现选择物化去重表，再在 ADR 中说明。
- **侧栏与真实能力**：当前 Web 侧栏是「能力模块」集合，与 **阶段 1 主漏斗**（Topic → Query → 采集 → 原始库 → 清洗 → 分析 → 看板/报告）尚未一一对应；**Content** 页仍偏「全库浏览」占位；**Analytics / Reports** 多为静态占位。用户会感到「每一步割裂」——需在导航或引导流上显式对齐 CONTEXT 已定的优先级（先 Topic 级原始库）。

## Resolved decisions

- **阶段 1 浏览优先级（2026-05）**：采集成功后，产品侧优先交付 **原始内容库（Raw content 列表）**，再推进清洗 / AI / 聚合看板。
- **原始内容库默认范围（2026-05）**：列表默认以当前 **Topic** 为范围，聚合其下所有 **Query** 产生的 **Raw content**；**Query** 作为筛选项收窄范围（不默认锁单条 Query）。
- **跨 Query 同帖（2026-05）**：默认 **合并展示**（一行 + 命中 Query 信息）；提供 **「按 Query 展开重复行」** 开关，便于对比各 Query 命中情况（对应选项 B3）。
- **原始内容列表默认排序（2026-05）**：以 **帖子发表时间**（`published_at`）新在前为主；若缺少发表时间，则退回 **入库/抓取时间**（`captured_at`）（对应选项 C2）。
- **原始内容列表行必显（2026-05）**：Topic 合并视图下每行至少包含：**平台**、**发表时间**（无则显示抓取时间）、**正文摘要**、**外链**、**作者**（handle 或显示名）、**命中的 Query 名称**（可多选合并展示）（对应选项 D3）。
- **降低「步骤割裂」的产品策略（2026-05）**：采用 **E2** —— **保留 Topics / Queries / Tasks / Content 等多页结构**，在全应用增加与阶段 1 漏斗一致的 **流程提示（步骤条或等价）**；在 **Crawl task 成功**等关键节点提供 **深链**，跳转到 **当前 Topic 下的原始内容库**（而非孤立的全库页）。不采用单页大向导（非 E1）、不单靠文档清单（非 E3）。
